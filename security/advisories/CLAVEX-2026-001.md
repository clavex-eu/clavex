## Summary

During the implementation of the CIBA (Client-Initiated Backchannel Authentication)
endpoint, an intermediate version of the `BackchannelAuthorize` handler logged the
raw subject identifier (`sub`) extracted from the `id_token_hint` parameter at the
`Debug` log level using zerolog's structured output.

Although debug logging is disabled in production (`CLAVEX_DEV=false` is the default),
an operator who enabled verbose logging for diagnostic purposes — or a deployment
misconfigured with `CLAVEX_DEV=true` in production — could have had user subject
identifiers written to log aggregation systems. Depending on the logging infrastructure,
this could expose user identifiers to personnel who do not have a legitimate need for
them, violating least-privilege access to personal data.

---

## Affected Code

**File**: `internal/handler/ciba.go`  
**Function**: `BackchannelAuthorize()`  
**Commit range**: `44569fc` (initial CIBA implementation) through the fix commit

The vulnerable log statement (intermediate development version, reconstructed):

```go
// BEFORE (vulnerable — never shipped in a release)
log.Debug().
    Str("login_hint_sub", sub).   // <-- exposes user identifier
    Str("client_id", clientID).
    Msg("ciba: extracted sub from id_token_hint")
```

---

## Fix

The `sub` value is no longer written to any log output. The corrected code:

1. Silently passes the extracted UUID to the resolver without logging it.
2. The final `Info`-level log at the end of `BackchannelAuthorize` uses only
   the `auth_req_id` (a random UUID with no user-identifying information)
   and the `client_id`.

The production log statement (safe):

```go
// AFTER (safe)
log.Info().
    Str("auth_req_id", authReqID).
    Str("client_id", clientID).
    Str("org", orgSlug).
    Str("binding_message", bindingMessage).
    Msg("ciba: backchannel auth request accepted")
```

---

## Impact

- **Confidentiality**: Low — subject identifiers (UUIDs in Clavex) could be written
  to logs accessible to operators. No authentication bypass or session theft.
- **Integrity**: None — the vulnerability is read-only.
- **Availability**: None.

**Exploitability**: Requires access to application logs (i.e. operator-level access).
An external attacker cannot trigger log exposure without pre-existing server access.

---

## Mitigating Factors

1. **Never shipped**: The vulnerable code existed only in pre-release development
   commits and was identified and removed before any tagged release (`v0.x.y`).
2. **Debug-level only**: The statement used `log.Debug()`, which is suppressed in
   production unless `CLAVEX_DEV=true` is set. The production hardening guide
   explicitly states not to set this flag in production.
3. **No user enumeration**: The sub values logged are Clavex-internal UUIDs, not
   email addresses or usernames.

## Recommendations for Operators

If you are running a pre-release build from the `main` branch between commits
`44569fc` and the fix commit, and you have `CLAVEX_DEV=true` in your environment:

1. Rotate or purge log data collected during that period.
2. Update to the latest `main` commit or await the next tagged release.
3. Ensure `CLAVEX_DEV=false` (or the env var absent) in all production deployments.
